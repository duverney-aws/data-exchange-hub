Authentication

All API endpoints require a valid Cognito JWT token in the Authorization header.

Cognito Configuration

User Pool IDus-east-1_uT79YqQ5x

Hosted UI Domainpharma-data-exchange-portal.auth.us-east-1.amazoncognito.com
Sign-in AliasEmail
Groupsmerck-admins, cmo-users
Custom Attributescustom:organization — links CMO user to their cmoId

Auth Flow

  1. User visits portal → redirected to Cognito Hosted UI
  2. User signs in with email/password → Cognito returns authorization code
  3. Frontend exchanges code for JWT tokens (ID token, access token, refresh token)
  4. All API calls include: Authorization: Bearer {idToken}
  5. API Gateway validates JWT via Cognito Authorizer → passes claims to Lambda

JWT Claims Used by Backend

{
  "sub": "user-uuid",
  "email": "user@example.com",
  "cognito:groups": ["merck-admins"],
  "custom:organization": "cmo-xxxxxxxx",
  "given_name": "John",
  "family_name": "Doe"
}
Role Enforcement: The backend reads cognito:groups to determine the user's role. CMO users are filtered to their own organization using custom:organization — this is enforced server-side and cannot be bypassed by modifying query parameters.

Example: Obtain a Token (curl)

# Initiate auth to get tokens
aws cognito-idp initiate-auth \
  --client-id 2eitutbi1gkogudchaudb9rtr4 \
  --auth-flow USER_PASSWORD_AUTH \
  --auth-parameters USERNAME=merck-admin@pharma-exchange.demo,PASSWORD=MerckAdmin2026! \
  --region us-east-1

# Use the IdToken from the response:
TOKEN="eyJraWQiOi..."

# Make an API call
curl -H "Authorization: Bearer $TOKEN" \
  https://4841ud8xri.execute-api.us-east-1.amazonaws.com/prod/api/cmo
